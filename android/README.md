# Android App

Experimental Android control panel and foreground-service host for the WebRTC tunnel, offer mode only.

- **Min SDK**: 26 (Android 8.0 Oreo)
- **Target SDK**: 35
- **Version**: 0.3.2
- **App ID**: `com.phillipchin.webrtctunnel`
- **Language**: Kotlin 2.0 + Jetpack Compose + Material 3

## Status

Offer mode only. Answer mode is disabled in this release. The app runs the Rust tunnel runtime (`p2p-mobile`) inside a foreground service so the tunnel persists while the app is backgrounded.

## Prerequisites

### Rust toolchain

```bash
# Add Android targets
rustup target add aarch64-linux-android x86_64-linux-android

# Install cargo-ndk
cargo install cargo-ndk
```

### Android SDK

Android NDK must be installed (via Android Studio SDK Manager or standalone).

## Building the JNI libraries

The Gradle `preBuild` task automatically calls `buildRustAndroid`, which compiles `crates/p2p-mobile` via `cargo-ndk` and copies the resulting `.so` files into `app/src/main/jniLibs/`.

Run it explicitly from the repository root:

```bash
cargo ndk -t arm64-v8a -t x86_64 -o android/app/src/main/jniLibs build -p p2p-mobile --release
```

Or via Gradle from the `android/` directory:

```bash
cd android
./gradlew buildRustAndroid
```

Supported ABIs: `arm64-v8a`, `x86_64`.

## Building the APK

From the `android/` directory:

```bash
./gradlew assembleDebug
```

The debug APK is written to `app/build/outputs/apk/debug/app-debug.apk`.

## Installing on a device

```bash
./gradlew installDebug
```

Or with `adb` directly:

```bash
adb install -r -d app/build/outputs/apk/debug/app-debug.apk
```

## Running tests

```bash
# Lint
./gradlew lintDebug

# Unit tests (JVM, no device required)
./gradlew testDebugUnitTest

# Instrumentation tests (requires attached device or running emulator)
./gradlew connectedDebugAndroidTest
```

All three commands should pass before merging any Android change. Required CI
only runs `testDebugUnitTest` (via Robolectric) â€” it does not run
`connectedDebugAndroidTest`, since that needs a device/emulator CI does not
provision. Critical truthfulness behavior (e.g. `TunnelForegroundService`
stop-failure handling) therefore has Robolectric coverage under `src/test` as
the source of required-CI proof; the `androidTest` instrumentation suite
remains supplemental platform-integration coverage on top of that, not a
substitute for it.

## Network policy

Cellular and metered Wi-Fi tunnels are **blocked by default**. The user must explicitly enable `Allow metered connections` in Settings, or tap `Allow for this session` on the Home screen to override once. When the device moves back to unmetered Wi-Fi, the tunnel automatically resumes if `Resume on unmetered Wi-Fi` is enabled (default: on).

## Connection (ICE) mode

`Settings â†’ Advanced â†’ Connection (ICE) mode` lets you choose how ICE candidates
are gathered, written into the generated config as `[webrtc].android_ice_mode`:

| Mode | When to use |
|---|---|
| `native` | Reach a peer on a **different** network (over the internet). Gathers server-reflexive (srflx) candidates via STUN, so it can traverse NAT. |
| `vnet_mux` | Reach a peer on the **same** Wi-Fi. Binds a `0.0.0.0` UDP mux and advertises the device's real local IPv4 (injected from `ConnectivityManager`) as the host candidate. |

The choice takes effect the next time the tunnel is started; it rewrites only the
`android_ice_mode` line in the active config, so saved broker/peer/forward settings
and the identity key are preserved. New installs default to `vnet_mux` (the strict,
proven Android path); switch to `native` for cross-network peers.

## Identity security

The private Ed25519/X25519 identity is stored encrypted using AES-256-GCM with a key managed by the Android Keystore. The key is bound to the device and never leaves it. The plaintext identity is only materialised in memory immediately before the native tunnel starts, and it is never logged or written to an accessible path in plaintext.
